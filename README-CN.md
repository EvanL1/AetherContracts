# AetherContracts

[English](README.md)

**智能原生物理世界的类型化信任边界。**

AetherContracts 是 AetherEdge、AetherCloud 和独立实现之间公开、语言无关的互操作权威。规范定义语义，数据结构规范定义结构，样例固定具体表现，一致性测试套件提供可执行证据。任何语言绑定都不能成为第二套事实来源。

在 AetherIoT 的产品方向中，智能体不能自行编造设备名称、能力、命令或消息结构。它只能依据已声明的契约进行推理，并生成可在接触物理设备前验证的带版本方案。

当前版本是 **0.1.0-alpha.3**，提供实验性的物模型与 CloudLink 契约。它尚未定义完整的用户意图、方案、策略、模拟或家庭自动化语言，也不是可用于生产环境切换的 CloudLink 版本。

## 在智能原生闭环中的职责

```text
用户意图
    ↓
智能体发现已声明的真实能力
    ↓
智能体生成类型明确的方案
    ↓
AetherContracts 验证结构与兼容性
    ↓
云端治理与边缘端本地验收
    ↓
确定性执行
```

契约把模型输出变成可以检查的数据，而不是可以直接执行的权力。系统可以在边缘端评估本地策略之前，拒绝虚构能力、不兼容版本、无效状态迁移和结构不安全的消息。

## 规划中的契约族

下列内容代表产品方向，在 alpha.3 中尚不具备规范效力：

- 空间、成员、偏好和环境上下文语义；
- 用户意图、约束、持续时间和置信度；
- 生成方案、解释、模拟和确认；
- 策略、风险、权限、审计和回退证据；
- 已交付的自动化版本及其观测结果；
- 智能体反馈可以建议修订，但不能静默改变正在运行的系统。

每个新契约族都必须形成完整的纵向切片：规范语义、封闭结构、有效与无效样例、兼容性规则和可执行的一致性测试证据。

## 安全边界

- 物模型能力只是声明，不是授权。
- 所有能力默认拒绝。
- CloudLink 不是任意远程调用，不提供直接寄存器或物理控制操作。
- 受治理的任务仍需通过云端策略和边缘端最终决策。
- MQTT 的发布确认只是传输证据，不能证明云端业务事实已持久提交。
- 智能体输出在通过契约和应用验证前一律视为不可信输入。

边缘端始终对实时点位状态、确定性规则、安全联锁和物理执行负责。

## 仓库结构

- spec/：具有规范效力的英文语义和生命周期规则。
- schemas/：封闭的数据结构定义。
- profiles/：传输和标准对齐配置。
- fixtures/：有效、无效、上下文、迁移和黄金样例。
- compatibility/：失败分类和兼容性门槛。
- tck/：语言无关场景和仓库契约测试。
- packages/：实验性语言绑定，不具备规范效力。
- contract-manifest.json：版本身份和制品哈希。

运行自包含的契约检查：

```sh
pnpm test:tck
```

运行全部语言绑定与打包检查：

```sh
pnpm check
```

默认验证流程不需要消息代理、数据库、云账号或设备。GitHub 标签、发布包、公开的 SHA-256 校验值和精确版本清单共同构成源码分发路径。消费者应提交封闭依赖锁，并在离线状态下验证导入内容。
