# AetherContracts

[English](README.md)

**文档网站：** [docs.aetheriot.dev/aethercontracts](https://docs.aetheriot.dev/aethercontracts/)

**智能原生物理世界的类型化信任边界。**

AetherContracts 是 AetherEdge、AetherCloud 和独立实现之间公开、语言无关的互操作权威。规范定义语义，数据结构规范定义结构，样例固定具体表现，一致性测试套件提供可执行证据。任何语言绑定都不能成为第二套事实来源。

在 AetherIoT 的产品方向中，智能体不能自行编造设备名称、能力、命令或消息结构。它只能依据已声明的契约进行推理，并生成可在接触物理设备前验证的带版本方案。

最新发布版本是 **v0.1.0-alpha.3**。当前源码面向尚未发布的
**0.1.0-alpha.4** 开发目标，提供实验性的物模型、外部设备集成与 CloudLink
契约。两者都不是可用于生产环境切换的 CloudLink 版本。

## 在智能原生闭环中的职责

```text
用户意图
    ↓
智能体发现已声明的真实能力
    ↓
智能体生成类型明确的方案
    ↓
AetherContracts 验证结构与兼容性
    ↓
云端治理与边缘端本地验收
    ↓
确定性执行
```

契约把模型输出变成可以检查的数据，而不是可以直接执行的权力。系统可以在边缘端评估本地策略之前，拒绝虚构能力、不兼容版本、无效状态迁移和结构不安全的消息。

## 规划中的契约族

下列内容代表产品方向，在 alpha.4 中尚不具备规范效力：

- 空间、成员、偏好和环境上下文语义；
- 用户意图、约束、持续时间和置信度；
- 生成方案、解释、模拟和确认；
- 策略、风险、权限、审计和回退证据；
- 已交付的自动化版本及其观测结果；
- 智能体反馈可以建议修订，但不能静默改变正在运行的系统。

每个新契约族都必须形成完整的纵向切片：规范语义、封闭结构、有效与无效样例、兼容性规则和可执行的一致性测试证据。

## 安全边界

- 物模型能力只是声明，不是授权。
- 所有能力默认拒绝。
- CloudLink 核心不是任意远程调用，不提供直接寄存器或物理控制操作。
- 独立的外部设备控制扩展默认关闭，首版只允许受治理的设备电源设定动作；公开消息不能携带调用方指定的家庭助理操作、服务参数、地址、令牌或任意数据对象。
- 边缘端策略保留最终执行权；家庭助理接受请求不等于物理设备已经完成动作。
- 外部设备集成只传递规范化拓扑和观测，不包含服务地址、令牌、任意属性集合或直接服务调用。
- 外部设备集成通过 CloudLink 上报时，网关身份来自外层认证会话；拓扑和观测载荷保持原样，旧版消费者不会被默认为兼容。
- 家庭助理连接信息只保存在边缘端；其状态事件也不能单独证明物理设备已经完成动作。
- 受治理的任务仍需通过云端策略和边缘端最终决策。
- MQTT 的发布确认只是传输证据，不能证明云端业务事实已持久提交；累计持久确认也不得越过未持久且未声明丢失的位置。
- 网关签名认证提案先在既有的上行会话主题发送受限频的挑战请求。请求只会触发对已委任且凭据有效的网关进行查验，本身不是身份证明。云端必须先持久化挑战再发布；重试时发布完全相同的挑战，并在接受会话时原子地一次性消费挑战。该提案默认关闭。
- 网关签名的交付消息和心跳使用两套精确签名映射。心跳的新鲜度绑定已接受会话协商的间隔和显式评估时间；完全相同的重放不会刷新在线状态。
- 每个已持久交付位置只保存一个与会话无关的不可变摘要，以及一个当前会话绑定。同一会话的精确重放不会重复业务副作用；只有会话世代严格增加且属于当前有效会话时，才可重新绑定并补发当前会话的确认。相同世代更换会话标识、世代回退或修改凭据代次、发送时间、过期时间、消息类型、批次和业务摘要都会被拒绝。
- 只有与业务副作用原子持久提交的重放记录才能在原过期时间之后补发确认。仅完成鉴权、预留中或尚未提交的记录不能绕过首次过期门槛；重新绑定也必须先持久化，之后才能补发确认。
- MQTT 前缀是授权命名空间。一个命名空间内的网关标识必须唯一；多租户部署必须按租户和项目隔离前缀，或使用云端全局唯一的网关标识。消息代理主体只能访问精确命名空间和网关主题，前缀与主题本身都不能证明网关身份。云端必须先按可信命名空间划分持久重放记录，再使用不变的四字段线协议身份查询；禁止跨命名空间复用记录，也禁止缺少命名空间条件的全局查询。
- alpha.4 的 `session-accepted` 没有签名，也不携带 `challenge_id` 或 `client_nonce`，因此协议层无法排除另一轮握手的延迟或重放响应。带签名且关联握手的响应属于下一协议版本门槛；未签名的持久应用确认同样阻断生产认证。
- 智能体输出在通过契约和应用验证前一律视为不可信输入。

边缘端始终对实时点位状态、确定性规则、安全联锁和物理执行负责。

## 仓库结构

- spec/：具有规范效力的英文语义和生命周期规则。
- schemas/：封闭的数据结构定义。
- profiles/：传输和标准对齐配置。
- fixtures/：有效、无效、上下文、迁移和黄金样例。
- compatibility/：失败分类和兼容性门槛。
- tck/：语言无关场景和仓库契约测试。
- packages/：实验性语言绑定，不具备规范效力。
- contract-manifest.json：版本身份和制品哈希。

运行自包含的契约检查：

```sh
pnpm test:tck
```

运行全部语言绑定与打包检查：

```sh
pnpm check
```

默认验证流程不需要消息代理、数据库、云账号或设备。GitHub 标签、发布包、公开的 SHA-256 校验值和精确版本清单共同构成源码分发路径。消费者应提交封闭依赖锁，并在离线状态下验证导入内容。
